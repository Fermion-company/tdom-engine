# 第5章 旧バックエンド — v1（独立コンパイル）とv0（内部エンジン）

2つの旧世代は単なる遺物ではありません。v1はfork(2)が使えない環境
（例：一部のコンテナ・Windows直載せ）での代替であり、v0はTeXすら無い
環境のフォールバック兼、アーキテクチャの参照実装です。また共有基盤
（ブロック分割・差分・ページ再利用）は3世代とも同じコードを使って
います。

## 5.1 共有基盤

### segmenter.js — Source DOMの根

全世代の増分性はここから始まります。`segmentBody()` は本文を1回の
線形走査で**ブロック**に割ります。境界は：

- グループ深さ0・環境深さ0での**空行**
- `\section` / `\subsection` / `\begin{...}` / `\[` で始まる行の直前
- 環境やディスプレイ数式が閉じた行の直後

環境内の空行では割らない（itemize内の空行等）ため、環境は丸ごと
1ブロックになります。各ブロックはFNV-1aの内容ハッシュを持ちます。

`diffBlocks()` は旧ブロック配列と新分割結果をハッシュで突き合わせ、
共通prefix・共通suffixのブロックを**オブジェクトごと**再利用します
（これが全キャッシュの生存機構——JSのオブジェクト同一性がそのまま
「変わっていない」の証明になる）。中間の変更ブロックはidを保って
作り直し、追加分には新idを振ります。

### source-store.js / hash.js / page.js

範囲編集つきテキストバッファ（行列⇔オフセット変換つき）、FNV-1a、
v0/v1用の単純ページ分割（v3は第3章のpagebuilderを使用）。

## 5.2 v1: lualatexバックエンド（engine-lua.js + engine/luatex/）

v3の前段階で、**「編集のたびに新しいlualatexプロセスを起動するが、
コンパイルするのはdirtyブロックだけ」**という設計です。1編集
約430ms（プロセス起動が支配的）。仕組みの要点：

### ブロック独立コンパイルと状態注入

dirtyなブロック群を1つのdriver文書にまとめ、ブロックごとに

```tex
\setcounter{section}{2}\setcounter{equation}{3}...  % 入口状態を注入
\global\@namedef{r@sec:intro}{{1}{1}}...            % 既知ラベルを注入
\setbox\TDOMbox=\vbox{...ブロック本文...}
（linesplit.luaがガレー抽出）
\shipout\box255                                      % タイトページとして出荷
（カウンタ出口値をLuaで記録）
```

を実行します。出力PDFの第kページ＝第kブロックで、`pdftocairo` で
ページごとにSVG化してブロック画像（チャンク）にします。v3と違い
プレビューは**ブロック単位の画像**です。

### デルタ方式のカウンタ連鎖

独立コンパイルでは各ブロックが見るカウンタは注入値です。ブロックの
効果を**絶対値でなく差分（delta = 出口 − 入口）**で記録することで、
「入口がどうであれ、このブロックは section を+1 する」という形で
連鎖を合成可能にしました（絶対値記録だと、孤立実行時の値で連鎖が
壊れる——第6章・罠に相当する初期バグ）。文書全体の初回ビルドだけは
注入なしで自然連鎖させ、実測の入口/出口を記録します。

### 実ラベルはauxから

出荷時に書かれる `.aux` を `\tdomblockmark` マーカーで区切って
パースし、**TeX自身が計算したラベル値**を取得します（自前で番号を
数えない）。

### プリアンブルの事前コンパイルとスモークテスト

`mylatexformat` でプリアンブルを `.fmt` にダンプすると起動が縮み
ますが、**TikZを含むformatはPDFバックエンドを壊す**既知問題が
あります（「cannot find referenced object」）。そこでダンプ後に
必ず1ページ出荷するプローブを流し、失敗したらformatを捨てて
プリアンブル逐次読み込みに自動フォールバックします
（`backend.js` `prepare()`／`#tryProbe()`）。

### キャッシュ

コンパイル結果は `(プリアンブルhash, ブロックhash, 入口状態,
参照ラベル値, noindent)` をキーに記憶され、**undoは0コンパイル**
（1ms）になります。チャンクは内容アドレス（SVG＋メトリクスのhash）で
重複排除され、「空白だけの編集→出力バイト一致→パッチゼロ」が
成立します。

## 5.3 v0: internalバックエンド（依存ゼロの自前組版）

TeXが無い環境でも全アーキテクチャが動くことを示す参照実装で、
プロジェクト最初の1日に書かれたものです。約1,800行。品質はデモ水準
（貪欲行分割・自前の簡易数式）ですが、**「常駐＋差分＋最小パッチ」の
全配管はv3と同じ**です。約1ms/編集。

| モジュール | 内容 |
|---|---|
| `tokenizer.js` | 標準catcode前提のTeXトークナイザ（dirtyブロックのみ再走査） |
| `macro-vm.js` | `\newcommand`走査とトークン列展開。**どのマクロを消費したかを記録**し、マクロ再定義時に利用ブロックだけを無効化 |
| `semantic.js` | 展開済みトークン→意味ノード（section/paragraph/list/displaymath、ラベル・参照の記録） |
| `metrics.js` | Times系AFM幅表の埋め込み（エンジンが組版の真実を持つため） |
| `layout.js` | グルー/ボックスモデルでの両端揃え行分割。段落・見出し・リスト・CJK文字の1字ボックス化 |
| `math-layout.js` | 上付き/下付き/分数/ギリシャ文字/演算子間隔の簡易数式 |
| `page.js` | 行ユニット流のページ分割（参照同一性による再利用は全世代共通の発明） |
| `display-list.js` | グリフ座標→描画命令（v3のglyphsコマンドの原型） |
| `pdf.js` | 手書きPDF 1.4ライタ（コア14フォント＋Symbol、xref手組み） |
| `engine.js` | 統括。成功条件レポート（dirtySourceNodes→…→patches）の原型 |

v0で確立され、そのままv3まで生き残った設計が3つあります：
**(1)** ブロック＝キャッシュ単位とハッシュ差分、
**(2)** 「レイアウトキー（内容＋文脈のhash）が一致すれば再計算しない」
という宣言的キャッシュ、
**(3)** ページ再利用を「行オブジェクトの参照同一性」で判定する手法。
